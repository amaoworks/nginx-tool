# Nginx-Tools TUI 设计文档

## 一、概述

将现有的 `nginx-site.sh` + `nginx-monitor.sh` 命令行工具整合为单一 TUI（终端交互界面），保留全部功能的同时提供更直观的操作体验。

**核心设计原则：**
1. 站点管理是站点级操作的中枢——新建、编辑、启用、停用、删除、单站点证书申请、日志跳转都从这里发起。
2. **弹窗仅用于高危操作确认**（删除站点、还原备份、重启服务等），编辑类操作采用主视图覆盖模式，仅替换右侧主视图区域，保持界面统一性。
3. 所有主视图覆盖模式都显示清晰的路径导航，支持 `Esc` 返回上级。

## 二、界面布局

```
┌────────────────────────────────────────────────────────────┐

├──────────────┬─────────────────────────────────────────────┤

│  📁 站点管理 │                                             │
│  🔐 证书管理 │  · 仪表盘 → 状态概览                          │
│  📋 日志查看 │  · 站点管理 → 站点表格 + 新建 + 编辑 + 操作     │
│  ⚙️  服务控制 │  · 证书管理 → 证书操作面板                     │
│  💾 备份还原 │  · 日志查看 → 实时日志流                       │
│              │  · 服务控制 → 服务操作按钮                     │

### 尺寸比例

| 区域       | 宽度比例 | 说明                         |
| ---------- | -------- | ---------------------------- |
| 左侧边栏   | 25%      | 菜单导航，固定宽度不小于 20 列 |
| 右侧主视图 | 75%      | 内容展示与交互区域            |
| 顶部标题栏 | 100%     | 固定 1 行                    |
| 底部状态栏 | 100%     | 固定 1 行，快捷键根据上下文动态变化 |

### 主视图覆盖原则

所有“覆盖模式”均指**右侧主视图区域被完整替换**，不是整个应用全屏接管。

- 左侧边栏保持可见，当前一级菜单不跳变
- 顶部标题栏保持可见，持续展示全局状态
- 底部状态栏保持可见，仅上下文快捷键变化
- 新建站点、编辑站点、原始配置编辑、证书子流程都必须遵循这个原则

这样可以确保用户在深层操作时，仍然保留全局导航、运行状态和返回路径，不会丢失上下文。

### 焦点指示规范

为确保用户清楚当前交互位置，所有可交互元素必须有明确的焦点状态：

| 元素类型     | 未聚焦状态          | 聚焦状态                |
| ------------ | ------------------- | ----------------------- |
| 菜单项       | 空格 + 文字         | `▸` + 文字 + 反色背景    |
| 输入框       | `[________]`        | `[▸_______]` + 边框高亮  |
| 单选按钮     | `○` / `◉`          | `○` + 边框高亮           |
| 复选框       | `[ ]` / `[✕]`      | `[[ ]]` / `[[✕]]`       |
| 按钮         | `[  确定  ]`        | `[[  确定  ]]`           |
| 表格行       | 普通显示            | 整行反色背景             |
| 下拉选择     | `[选项   ▼]`        | `[▸选项   ▼]` + 边框高亮 |
| 注入槽       | 灰色边框            | 青色边框 + 内部光标       |

**焦点切换规则：**
- `Tab` / `Shift+Tab`：仅用于在焦点区之间正向/反向切换
- `↑↓`：仅用于当前纵向列表、表单字段或模板列表内移动
- `←→`：仅用于同一行的可选项之间切换（如单选组、分段控件、按钮组、标签页）
- `Space`：切换复选框、开关或当前聚焦的二值选项
- `Enter`：激活当前项、确认当前项或提交当前操作
- `Esc`：返回上一级或取消当前操作
- 不允许在普通页面中让 `Tab` 与 `↑↓` 表达同义动作
- 不允许不同顶层页面重新定义 `Tab` 的根语义

### 交互状态分层

整个 TUI 的交互只分为以下五类状态：

1. **顶层浏览态**：仪表盘、站点列表、证书、日志、服务、备份的正常浏览状态。
2. **页面子区域浏览态**：页面内部多个可切换区域之间的浏览状态。
3. **文本输入态**：单行输入框、多行编辑器、搜索框等接收字面输入的状态。
4. **操作条态**：焦点位于保存、重置、返回等按钮组时的状态。
5. **弹窗/帮助态**：确认弹窗、帮助弹窗激活时的状态。

所有页面都必须在这五类状态内定义自己的交互，不允许额外引入未声明的按键模式。

### 按键分发优先级

按键处理按以下优先级分发：

1. 弹窗/帮助态
2. 文本输入态
3. 页面子区域浏览态
4. 顶层浏览态
5. 全局快捷键

如果当前处于文本输入态，普通字符必须优先作为输入内容，不得再触发页面动作、模式切换或菜单跳转。

`Esc` 是唯一允许跨层退出的通用按键，但其语义始终是“退一层”，不得在同一层承担多个不相关动作。

### 输入态契约

文本输入态允许的按键：

- 普通字符
- `Backspace`
- `Delete`
- `Left` / `Right`
- `Enter`
- `Tab` / `Shift+Tab`
- `Esc`

多行编辑器可额外使用 `Up` / `Down` 在文本内部移动。

文本输入态禁止触发的动作：

- 顶层菜单跳转：`1` - `6`
- 全局退出：`q`
- 页面字母热键：如 `a`、`o`、`r`、`s`、`n`、`d`、`c`、`l`

输入态下，所有可打印字符都必须按字面写入当前输入目标。

### 底部状态栏与帮助视图规范

- 底部状态栏是全局唯一的动作提示源。
- 页面内部可以显示状态信息、刷新时间、只读提示、错误信息、选中项摘要，但不再显示完整快捷键列表。
- 底部状态栏每次最多显示 4-6 组与当前焦点最相关的操作。
- 使用 `?` 打开帮助视图，帮助视图分为三部分：
  - 全局快捷键
  - 当前页面快捷键
  - 当前焦点快捷键
- 帮助视图用于承载完整键位说明，状态栏不承担完整帮助文档职责。

### 路径导航规范

所有主视图覆盖模式（新建、编辑等）在标题栏显示完整路径，便于用户理解当前位置：

```
┌──────────────────────────────────────────────┐
│  📁 站点管理  ▸  新建站点                     │  ← 路径导航
│  ...                                          │
│  [Esc] 返回站点列表                           │  ← 返回提示
└──────────────────────────────────────────────┘
```

路径格式：`<一级菜单> ▸ <二级功能> ▸ <三级功能>`

## 三、弹窗设计规范

### 弹窗使用原则

**弹窗仅用于以下场景：**

| 场景类型 | 说明 | 示例 |
|----------|------|------|
| 高危操作确认 | 不可撤销或影响重大的操作 | 删除站点、还原备份、重启服务 |
| 操作结果提示 | 短暂显示后自动消失 | 操作成功/失败提示 |
| 必要信息输入 | 操作前必须获取用户输入 | 申请证书时确认站点 |

**禁止使用弹窗的场景：**

| 场景类型 | 应采用的方式 |
|----------|--------------|
| 内容编辑 | 主视图覆盖编辑模式 |
| 表单填写 | 主视图覆盖表单模式 |
| 配置修改 | 原地展开编辑区域 |
| 日志查看 | 切换到日志视图 |

### 确认弹窗（仅用于高危操作）

```
┌──────────────────────────────┐
│  ⚠️  确认删除                │
│                              │
│  即将永久删除站点: app        │
│  此操作不可撤销               │
│                              │
│  [[ 确认 ]]    [ 取消 ]      │
└──────────────────────────────┘
```

**交互：**
- `Tab` / `←→` 在按钮间切换
- `Enter` 执行选中按钮
- `Esc` 等同于"取消"
- 默认聚焦"取消"按钮（防止误操作）

### 操作结果提示

短暂显示（成功 2 秒，失败 3 秒）后自动消失，不阻断用户操作：

```
  ✓ 站点 app 已启用
  ✗ 配置测试未通过：unknown directive "proxy_pas" at line 15
```

**显示位置：** 主视图底部，不遮挡主要内容

### 视图 2：站点管理

该视图整合了原 `nginx-site.sh` 的 `list`、`new`、`edit`、`enable`、`disable`、`delete` 六个命令，是日常工作最核心的操作界面。

站点管理分为三层编辑深度，通过快捷键切换：

- **托管模式（默认）**：面向日常运维，一键新建、一键维护。用户只操作业务化字段和关键开关，不直接感知 Nginx 注入槽结构。
- **高级模式**：在托管模式基础上暴露更细的模板增强和配置预览，适合需要扩展但仍希望由工具维护主体结构的用户。
- **原始模式**：直接编辑配置文件全文，适合深度自定义。进入此模式后，用户需要自行理解配置结构，工具只提供保存、测试、回滚保护。

从页面组织上看，站点管理仍然保留“列表 / 新建 / 编辑 / 原始配置”四个覆盖视图，但新建与编辑默认都首先进入托管模式，高级模式是托管模式下的扩展入口，而不是主流程。

#### 子模式 A：站点列表（默认）

进入站点管理后首先看到此视图。

```
┌──────────────────────────────────────────────────────┐
│  📁 站点列表                             [3 个站点]   │
│                                                      │
│  ┌─────────────────────────────────────────────────┐ │
│  │ 状态  │ 名称  │ 域名→目标            │ 类型 │ SSL │ │
│  ├───────┼──────┼───────────────────┼────┼─────┤ │
│  │ ● 启用│ app   │ app.example.com →  │ 代理 │ ✓ 67天│ │
│  │       │       │ 127.0.0.1:8080     │     │      │ │
│  │ ○ 停用│ blog  │ blog.example.com → │ 静态 │ ✗     │ │
│  │       │       │ /var/www/blog      │     │      │ │
│  │ ● 启用│ emby  │ emby.example.com → │ Emby │ ⚠ 23天│ │
│  │       │       │ 192.168.1.5:8096   │     │      │ │
│  └─────────────────────────────────────────────────┘ │
│                                                      │
│  选中: app  ▏ 类型=代理 ▏ 目标=127.0.0.1:8080 ▏ SSL=67天│
└──────────────────────────────────────────────────────┘
```

**焦点区：**
- `Sidebar`
- `SiteTable`

**交互：**
- `↑↓` 导航选择站点（当前行反色高亮）
- `Enter` 进入当前站点的编辑视图（默认进入托管模式）
- `s` 切换启用/停用状态（自动测试配置并重载）
- `n` 进入新建站点覆盖模式
- `d` 删除当前选中站点（弹出高危确认框）
- `c` 为当前选中站点的域名申请 SSL 证书（弹出确认框，显示域名供确认）
- `l` 跳转到日志视图，自动筛选该站点日志
- `Tab` 切换到左侧菜单

**数据来源：**
- 遍历 `/etc/nginx/sites-available/*.conf`
- 检查 `/etc/nginx/sites-enabled/` 是否存在同名符号链接判断启用状态
- 从配置文件中提取 `server_name`、`proxy_pass`、`root` 指令
- SSL 状态 → 交叉匹配 `certbot certificates` 输出与站点域名，计算到期剩余天数
- SSL 列状态标识：
  - ✓ 绿色：证书有效，剩余 > 30 天
  - ⚠ 黄色：证书即将到期，剩余 7-30 天
  - 🔴 红色：证书紧急，剩余 < 7 天
  - ✗ 灰色：无证书
- 类型判定：包含 `proxy_pass` + emby 特征 → Emby，包含 `root` → 静态，其余 → 代理

**底部状态栏：** `站点列表 | [↑↓] 选择 [Enter] 编辑 [n] 新建 [d] 删除 [?] 帮助`

#### 子模式 B：新建站点（主视图覆盖模式）

按 `n` 从列表视图进入，完整替换右侧主视图区域，左侧菜单与全局状态保持不变。

新建站点默认采用**托管模式**。托管模式下，用户不需要理解 Nginx 指令应该放在 `server`、`location` 还是其他位置，只需要填写站点基础信息并勾选少量业务化选项，工具负责生成对应的 Nginx 配置。

│           （仅允许字母、数字、连字符）                         │
│                                                              │

│           （如: app.example.com，可附加别名）                  │

│           （端口号 如 8080 / IP:端口 / http(s)://地址）        │

│                                                              │
│  关键选项:                                                    │

│    静态站点:                                                  │
│      [◉ 普通静态] [○ SPA 单页]                               │

│           ┌──────────┐                                       │
│           │   创  建  │                                       │
│           └──────────┘                                       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**焦点区：**
- `Sidebar`
- `FormFields`
- `ActionBar`

**交互：**
- `Tab` / `Shift+Tab`：在“基础信息区 → 关键选项区 → 提交区”之间正向/反向切换
- `↑↓`：在当前区域的字段或选项行之间移动
- `←→`：在单选组、分段控件、按钮组之间切换
- `Space`：切换复选框、开关状态
- `Enter`：
  - 在输入框上 → 激活当前字段
  - 在按钮上 → 提交表单
- `Esc`：返回列表视图
  - 如果已填写内容，弹出确认框询问是否放弃
  - 确认放弃则丢弃内容，取消则继续编辑

**字段与类型逻辑：**

- `反向代理` 是一个大类，不再细分为“AI API 类型”“WebSocket 类型”等多个站点类型
- `流式响应 / AI API`、`WebSocket`、`大请求体 / 上传`、`浏览器跨域 CORS`、`长超时后端` 都是 `反向代理` 下的能力选项
- `静态站点` 下的 `普通静态 / SPA 单页`、`静态资源缓存`、`敏感路径保护` 也采用同样的托管选项方式
- `Emby/Jellyfin` 保持专用类型，默认带上该类业务的基础代理配置，只暴露少量必要字段

**字段逻辑：**

| 字段 | 校验规则 | 错误提示 |
|------|----------|----------|
| 站点名称 | 仅允许 `a-z A-Z 0-9 -`，不能为空，不能与已有站点重名 | "名称只能包含字母、数字、连字符" / "站点名称已存在" |
| 域名 | 基本域名格式校验（允许通配符 `*.example.com`） | "域名格式无效" |
| 代理目标 | 纯端口 → 自动补全为 `127.0.0.1:端口`；支持 `IP:端口` 和 `http(s)://地址` | "目标地址格式无效" |

**动态显示：**
- 选择"静态站点"时，代理目标输入框自动隐藏
- 选择"反向代理"时，展示 API / WS / 上传 / CORS / 长超时等能力开关
- 选择"静态站点"时，展示站点模式和缓存/保护选项
- `Emby/Jellyfin` 类型默认收敛为少量必要字段，不显示通用反向代理的全部能力开关
- 代理目标输入框默认显示占位提示
- 默认不展示注入槽、模板片段、Nginx 结构位置等高级概念

**创建流程：**

执行顺序严格如下，任何一步失败立即进入回滚流程，后续步骤不再执行：

1. **校验阶段**：收集表单输入 → 验证所有字段
   - 验证失败 → 在对应字段下方显示红色错误提示，焦点跳到第一个错误字段，流程终止
   - 联动校验：勾选"申请 SSL 证书"时必须同时勾选"立即启用"，否则禁用该复选框并显示提示"申请证书需要先启用站点"
2. **渲染阶段**：根据站点类型选择基础模板，再根据托管选项叠加对应能力，使用 minijinja 完成变量替换
3. **写盘阶段**：
   1. 写入临时文件 `~/.local/ngtool/tmp/<site>.conf`
   2. 校验目标路径 `/etc/nginx/sites-available/<site>.conf` 不存在
   3. 原子复制到目标路径
4. **启用阶段**（仅当勾选"立即启用"）：
   1. 创建符号链接 `/etc/nginx/sites-enabled/<site>.conf`
   2. 执行 `nginx -t`
   3. 测试通过 → `systemctl reload nginx`
   4. 测试失败 → 进入回滚流程
5. **证书阶段**（仅当勾选"申请 SSL 证书"且第 4 步成功）：
   1. 解析站点配置中的 `server_name`，按全部域名一次性申请
   2. 调用 `certbot --nginx -d <domain1> [-d <domain2>...]`
   3. 流式显示 certbot 输出
   4. 失败时不回滚已创建的站点配置（已提示用户手动 `certbot revoke` 或在证书页重试）
6. **结果阶段**：显示成功/失败提示，询问"是否返回列表？[Enter] 返回 [e] 继续维护"

> **为何证书申请依赖立即启用：** `certbot --nginx` 通过修改正在运行的 nginx 配置完成 HTTP-01 验证，必须保证站点已写入 `sites-enabled` 且 nginx 已加载该配置。未启用即申请会导致 certbot 找不到匹配的 server 块。

**失败回滚机制：**

按"反向次序"严格回滚，确保不留下不一致状态：

```
回滚检查顺序（与创建顺序相反）：
1. 是否已 reload？ → reload 一旦成功无法撤销，但 nginx 会继续使用回滚后的配置
2. 是否已创建符号链接？ → 删除 /etc/nginx/sites-enabled/<name>.conf，再次 nginx -t（验证回滚后状态）
3. 是否已创建配置文件？ → 删除 /etc/nginx/sites-available/<name>.conf
4. 是否已申请证书？ → 不自动 revoke，提示用户：「证书已申请但站点回滚，请在证书页处理」

回滚完成后显示：
  ✗ 创建失败：<具体错误信息>
  已回滚步骤：<已撤销的步骤列表>
```

**托管模式原则：**

- 默认只展示用户能理解的业务化选项，不暴露注入槽、模板片段和 `server/location` 结构位置
- 保存时由工具根据站点类型和当前开关重新生成配置
- 托管模式适用于“继续维护工具创建的站点”这一主路径
- 如果站点缺少托管标记或检测到自定义配置偏离过大，页面顶部必须显式提示当前处于“兼容模式”，并建议改用高级模式或原始配置

**高级模式入口：**

- 对少量常见增强需求，进入高级模式后可以继续使用模板增强、注入槽或配置预览
- 高级模式是托管模式的扩展入口，不是默认入口
- 高级模式中仍然保留“工具托管主体 + 用户补充细节”的思路

### 模式切换规范

- 托管模式、高级模式、原始模式不再通过普通字母快捷键直接切换。
- 编辑视图顶部必须显示模式标签：`[托管] [高级] [原始]`
- 焦点位于模式标签区时，`←→` 切换标签，`Enter` 进入对应模式。
- 文本输入态下不允许任何模式切换热键抢占输入。

**托管表单字段逻辑：**

- 进入时自动解析配置文件，优先恢复托管标记记录的站点类型和能力开关
- 修改表单值 + 保存 → 使用对应模板重新生成配置文件
- 保存时执行 `nginx -t` 测试，失败则：
  1. 显示具体错误信息和行号
  2. 保持编辑状态，不返回列表
  3. 提供选项：[r] 重试保存 [x] 放弃修改

**高级模式（模板增强 / 注入槽）：**

高级模式下，用户可以在托管主体不变的前提下补充额外配置，例如：

- 安全响应头、Gzip、CORS
- 流式响应增强、WebSocket、超时
- 静态资源缓存、敏感路径保护
- 少量附加 `location` 块或错误页

高级模式保留注入槽和片段库，但其定位是：

- 面向理解 Nginx 结构的用户
- 用于覆盖工具暂未抽象成开关的少量场景
- 不是一键建站、一键维护的默认路径

在高级模式中：

- 默认仍保持右侧主视图覆盖，不使用弹窗编辑
- 注入槽可以使用原地编辑或全屏编辑
- 模板片段与当前槽位联动展示，减少插错位置的风险

**为何保留高级模式：**

- 托管模式无法覆盖所有 Nginx 定制场景
- 直接跳到原始配置对多数用户过重
- 因此高级模式承担“少量结构化增强”的角色，而原始模式继续作为最终兜底

                    ┌─────────────┐
                    │   TUI 界面   │
                    └──────┬──────┘
                           │
              ┌────────────┼────────────┐
              │            │            │

    ┌─────────┼─────┐      │      ┌─────┴─────┐

### 数据读取路径

| 数据项 | 命令/文件 |
|--------|-----------|
| Nginx 状态 | `systemctl is-active nginx`, `systemctl status nginx` |
| Nginx 版本 | `nginx -v` |
| 站点列表 | `/etc/nginx/sites-available/*.conf` + `/etc/nginx/sites-enabled/` |
| 站点配置 | `/etc/nginx/sites-available/<site>.conf` |
| SSL 证书 | `certbot certificates` 输出解析 |
| 系统资源 | `df -h`, `free -h` |
| 日志 | `/var/log/nginx/*.log` |
| 备份 | `~/nginx-backups/*.tar.gz` |
| 自动续签状态 | `systemctl list-timers certbot.timer` |

### 执行操作路径

| 操作 | 命令序列 |
|------|----------|
| 启用站点 | 创建符号链接 → `nginx -t` → `systemctl reload nginx` |
| 停用站点 | 删除符号链接 → `nginx -t` → `systemctl reload nginx` |
| 新建站点 | 收集参数 → 模板渲染 → 写入文件 → 可选启用 + 证书 |
| 编辑站点（托管） | 读取文件 → 提取托管参数 / 特性 → 修改 → 模板重渲染 → 写入 |
| 编辑站点（高级） | 读取文件 → 提取参数 + 注入槽 → 修改 → 模板重渲染 → 写入 |
| 编辑站点（原始） | 读取文件 → 编辑 → 写入 |
| 申请证书 | 选择已有站点 → 解析 `server_name` → `certbot --nginx -d <domain...>` |
| 续期证书 | `certbot renew` |

### 模板、托管配置与注入槽的处理流程

```
读取配置文件
  │
  ├── 优先识别托管标记（类型 + 特性）
  ├── 提取模板参数（域名、目标、协议、静态目录等）
  ├── 若进入高级模式，再提取注入槽内容
  └── 若标记不完整，降级到兼容模式或原始模式
       │
  用户修改托管参数、特性或高级增强
       │
  保存时：
  1. 取基础模板
  2. 根据托管参数与能力开关生成配置主体
  3. 若处于高级模式，在标记位置注入用户自定义内容
  4. 写入配置文件
  5. nginx -t 测试
  6. 测试通过 → systemctl reload nginx
     测试失败 → 显示错误，保持编辑状态
```

## 九、高级模式预制片段库

以下是预定义的 Nginx 配置片段，供用户在**高级模式**中快速插入。

这些片段的定位是：

- 用于补充工具尚未抽象成“托管开关”的少量配置
- 面向理解 Nginx 指令放置位置的用户
- **不属于默认建站流程，也不是一键维护的主入口**

### 片段 1：Gzip 压缩配置

```nginx
gzip on;
gzip_vary on;
gzip_min_length 1024;
gzip_types text/plain text/css application/json application/javascript text/xml application/xml;
gzip_comp_level 6;
```

### 片段 2：CORS 跨域响应头

```nginx
add_header Access-Control-Allow-Origin *;
add_header Access-Control-Allow-Methods "GET, POST, OPTIONS";
add_header Access-Control-Allow-Headers "Origin, Content-Type, Accept";
```

### 片段 3：静态资源缓存规则

```nginx
location ~* \.(jpg|jpeg|png|gif|ico|css|js|woff2)$ {
    expires 30d;
    add_header Cache-Control "public, immutable";
}
```

### 片段 4：WebSocket 升级头

```nginx
proxy_http_version 1.1;
proxy_set_header Upgrade $http_upgrade;
proxy_set_header Connection "upgrade";
proxy_read_timeout 86400;
```

### 片段 5：自定义请求头（模板）

```nginx
proxy_set_header X-Custom-Header "your-value";
add_header X-Server-Name "nginx";
```

### 片段 6：禁止特定路径访问

```nginx
location ~ /\.(ht|git|svn) {
    deny all;
    return 404;
}
```

### 片段 7：允许特定 IP 访问

```nginx
allow 192.168.1.0/24;
allow 10.0.0.0/8;
deny all;
```

### 片段 8：反向代理缓冲控制

```nginx
proxy_buffering on;
proxy_buffer_size 4k;
proxy_buffers 8 32k;
proxy_busy_buffers_size 64k;
```

## 十、错误处理规范

### 用户操作错误

| 错误场景 | 处理方式 |
|----------|----------|
| 必填字段为空 | 在字段下方显示红色提示，焦点跳转到该字段 |
| 字段格式错误 | 在字段下方显示格式要求，高亮错误部分 |
| 站点名称重复 | 显示错误提示："站点名称已存在，请使用其他名称" |
| 域名格式无效 | 显示示例："正确格式: app.example.com" |
| 站点无可申请域名 | 禁止发起证书申请，并提示先在站点配置中补全 `server_name` |

### 系统操作错误

| 错误场景 | 处理方式 |
|----------|----------|
| nginx -t 失败 | 在输出区显示完整错误信息和行号，保持编辑状态 |
| certbot 申请失败 | 显示 certbot 输出，提示可能原因（域名未解析、端口未开放等） |
| 证书存在但站点缺失 | 在证书列表标记为孤立证书，并提示用户手动核对或清理 |
| 自动续签检查失败 | 显示 timer / hook 缺失项，并提供修复建议 |
| 权限不足 | 显示红色提示："需要 root 权限执行此操作" |
| 文件不存在 | 显示提示并建议创建或返回列表 |

### 网络错误

| 错误场景 | 处理方式 |
|----------|----------|
| certbot 连接失败 | 显示："无法连接 Let's Encrypt 服务器，请检查网络" |
| DNS 解析失败 | 显示："域名 DNS 解析失败，请确认域名已正确指向服务器" |
