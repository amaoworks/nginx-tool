# Nginx-Tools TUI 潜在纰漏与风险清单

## 一、文档定位

本文档只记录潜在纰漏、风险和后续处理建议，暂不修改 `design.md`。

风险等级：

| 等级 | 含义 |
|------|------|
| 高 | 可能导致配置损坏、服务中断、数据丢失或功能无法落地 |
| 中 | 可能导致功能不稳定、体验不一致或实现成本明显增加 |
| 低 | 可接受的边界问题，后续优化即可 |

## 二、高风险项

### R1：直接写 `/etc/nginx` 可能造成服务配置损坏

风险：

- 新建、编辑、还原过程中，如果写入后 `nginx -t` 失败，可能留下不可用配置。
- 停用站点后测试失败，如果未恢复符号链接，状态会与用户预期不一致。

建议：

- 所有写操作使用临时文件和回滚快照。
- 启用失败删除新符号链接。
- 停用失败恢复原符号链接。
- 编辑失败恢复原文件。
- 还原失败提示回滚到还原前备份。

状态：已在架构中约束，后续实现必须验证。

### R2：`certbot certificates` 文本解析不稳定

风险：

- certbot 输出格式随版本、语言环境、错误状态变化。
- 解析失败会影响证书列表和站点 SSL 状态。

建议：

- 解析失败不能影响站点列表。
- certbot 适配层返回 `Result<Vec<Certificate>, ParseWarning>`。
- 保留原始输出用于错误展示。
- 后续可研究读取 `/etc/letsencrypt/renewal/*.conf` 和证书文件过期时间作为更稳定来源。

状态：待实现时重点处理。

### R3：原始配置和托管表单模式互相覆盖

风险：

- 用户手写 Nginx 配置可能无法被表单解析。
- 表单保存可能覆盖未知自定义配置。

建议：

- Rust 生成的模板必须加入稳定托管标记和注入槽标记。
- 托管标记记录站点类型和能力开关，用于恢复“一键维护”表单。
- 注入槽标记统一格式：`# nginx-tools:custom-<slot>:start` / `:end`。
- 无标记配置进入兼容模式。
- 兼容模式下表单保存前必须提示风险。
- 原始配置模式永远保留。

状态：标记策略已在 `architecture.md §12.2-§12.4` 与 `design.md §四` 对齐统一，实现时按规范输出即可。

### R3A：托管配置与用户手工修改发生漂移

风险：

- 用户在原始模式或外部编辑器中修改了工具生成的配置，导致托管标记记录的能力开关与真实配置主体不一致。
- 再次进入托管模式时，如果继续按旧标记回填，可能生成与用户预期不符的新配置。

建议：

- 编辑站点时优先读取托管标记，但必须对关键字段和配置主体做一致性校验。
- 检测到明显漂移时，不再强行进入托管编辑，应降级为兼容模式或原始模式。
- UI 必须给出明确提示：`当前配置已偏离工具托管结构，建议使用高级模式或原始配置模式。`

状态：待架构实现时重点处理。

### R4：长任务阻塞 TUI

风险：

- certbot、renew、restore、tail 日志、systemctl status 可能阻塞输入和渲染。

建议：

- 所有长任务通过 tokio task 执行。
- 输出通过 channel 传回 UI。
- 任务状态可取消或至少可返回页面。

状态：架构已定义任务模型，实现时需严格遵守。

### R5：root 权限和只读模式边界不清

风险：

- 非 root 下误展示可执行快捷键。
- 用户触发写操作后才失败，体验差。
- 只读模式下 `$EDITOR` 可能仍可编辑临时文件但无法写回。

建议：

- 启动时确定 `RunMode`。
- UI 根据 `RunMode` 裁剪快捷键。
- 只读模式下写操作不进入执行器。
- 只读模式允许查看原始配置，但不允许保存。

状态：用户已确认允许只读模式。

### R6：还原备份可能覆盖非工具管理的 Nginx 配置

风险：

- `/etc/nginx` 可能包含用户手工维护的文件。
- 整目录覆盖可能删除或替换不应由工具管理的内容。

建议：

- **范围限定方案（首版采用）**：备份和还原范围只覆盖 `/etc/nginx/nginx.conf` + `/etc/nginx/sites-available/*.conf` + `/etc/nginx/sites-enabled` 启用关系（参见 `architecture.md §11.7.1`）。明确**不**覆盖 `conf.d/`、`snippets/`、`modules-enabled/`、`mime.types` 等内容。
- 备份包内必须含 manifest.toml，记录工具版本、范围、checksum；无 manifest 的备份只可查看不可还原。
- 还原前展示影响摘要：将覆盖哪些文件、将启用/停用哪些站点，让用户最后确认。
- 还原前必须自动创建 `source = "pre-restore"` 备份。
- 还原失败提供回到 pre-restore 备份的入口；二次失败保留临时目录人工干预。

状态：方案已落入架构文档，由 `execution.md P9` 闭环。

## 三、中风险项

### R7：固定 Debian/Ubuntu Nginx 目录结构

风险：

- CentOS、Alpine、Docker 镜像可能没有 `sites-available` / `sites-enabled`。
- 用户可能自定义 Nginx 根目录。

建议：

- 首版按已确认决策固定。
- 启动检测目录缺失时进入只读异常状态或显示不支持。
- 配置文件中预留路径字段，但 UI 暂不开放修改。

状态：用户已确认首版暂时固定。

### R8：首版只支持 systemd

风险：

- 非 systemd 环境无法 reload/restart/status。
- Docker 容器中通常没有 systemctl。

建议：

- 启动检测 `systemctl`。
- 缺失时禁用服务控制和依赖 reload 的写操作。
- 显示“当前环境不支持 systemd，首版暂不支持服务控制”。

状态：用户已确认首版只支持 systemd。

### R9：日志实时跟踪的实现方式

风险：

- 直接调用 `tail -F` 依赖系统工具。
- 自己实现文件跟踪需要处理 rotate、truncate、权限和编码。

建议：

- MVP 优先封装 `tail -F`。
- 后续再考虑纯 Rust 实现。
- 切换日志源时必须取消旧任务。

状态：用户已确认需要实时日志。

### R10：终端尺寸过小导致 UI 重叠

风险：

- 设计文档中的表格和表单较宽，小终端中可能无法完整显示。

建议：

- 设置最小可用尺寸提示。
- 表格列支持截断和横向信息摘要。
- 表单字段按可用宽度换行。
- 手工验收覆盖小窗口。

状态：待 UI 实现处理。

### R11：快捷键冲突

风险：

- `Enter` 在不同页面可能表示进入、提交或执行高风险操作。
- `r` 在仪表盘刷新，在证书页续签，在备份页还原。
- `Esc` 同时承担返回、取消、清除搜索。

建议：

- 快捷键解析必须绑定当前 `Route` 和焦点区域。
- 底部状态栏必须实时展示当前上下文。
- 对高危操作必须进入确认弹窗。
- 列表页的 `Enter` 应优先用于“进入编辑”而不是直接执行启停等状态变更操作。

状态：设计文档已有上下文状态栏，代码需落实。

### R11A：方向键语义不一致导致误操作

风险：

- 某些页面 `↑↓` 是切字段，某些页面又是切区域，用户容易丢失焦点。
- 某些页面 `←→` 有效，某些页面无反馈，导致“按了没反应”的体验。
- 若方向键语义未统一，托管模式的大量开关和分段控件会变得难用。

建议：

- 统一方向键规则：`↑↓` 负责当前纵向集合，`←→` 负责当前行可选项，`Tab` 负责跨区域。
- 不允许在普通表单中让 `↑↓` 与 `Tab` 完全同义。
- 所有分段控件、开关组、标签页都必须对 `←→` 给出明确反馈。

状态：待设计与实现统一落地。

### R12：`Ctrl+Enter` 终端兼容性

风险：

- 部分终端无法区分 `Enter` 和 `Ctrl+Enter`。

建议：

- 实现时检测 crossterm 是否能捕获。
- 如果不能捕获，不影响正常提交流程。
- 不把 `Ctrl+Enter` 作为唯一提交方式；如无明显收益，可直接取消该设计。

状态：待实现验证。

### R13：`$EDITOR` 外部编辑器与 TUI raw mode 冲突

风险：

- 未正确恢复终端会导致用户终端显示异常。
- 外部编辑器退出后文件状态可能变化。

建议：

- 调用外部编辑器前退出 raw mode 和 alternate screen。
- 编辑器退出后重新进入 TUI。
- 重新读取文件并提示是否检测到变化。

状态：用户已确认保留外部编辑器。

### R14：证书申请强绑定已有站点的边界

风险：

- 用户想为尚未创建的域名提前申请证书。
- 通配符证书、DNS challenge 不适合 `certbot --nginx` 流程。

建议：

- 首版保持绑定已有站点。
- 通配符和 DNS challenge 暂不支持。
- UI 明确提示“请先创建站点并配置 server_name”。

状态：用户已确认保持绑定已有站点。

### R15：Emby/Jellyfin 类型识别不可靠

风险：

- 仅根据 `proxy_pass` 和少量特征判断类型可能误判。

建议：

- Rust 模板生成时加入工具标记。
- 老配置只做启发式识别，允许显示为"代理"。
- 类型误判不能影响原始配置编辑。

状态：待模板实现处理。

### R19：CJK 双宽字符终端兼容性

风险：

- `tui-textarea` 与 ratatui 在历史上对 CJK 全角字符的光标定位、宽度计算、行尾换行存在 corner case，可能导致输入框光标错位、表格列宽溢出。
- `LANG=C` 环境下中文文案可能显示为乱码。

建议：

- 每次涉及输入或表格的代码改动后执行 `architecture.md §16.4` 定义的 CJK 冒烟测试矩阵。
- 启动时检测 locale，非 UTF-8 时给出提示但仍可运行。
- 严重 corner case 出现时不做临时补丁，评估替换为 `tui-input` + 自绘的备选方案。
- 表格列宽计算必须用 `unicode-width` 而非 `String::len()`。

状态：实现期重点验证；feasible 但需要冒烟测试守门。

## 四、低风险项

### R16：只做中文界面，后续国际化成本上升

风险：

- 文案散落在 UI 中，后续要双语时改动较大。

建议：

- 首版可直接中文。
- 代码中尽量集中常用状态文案。

状态：用户已确认只做中文。

### R17：配置路径使用 `~/.local/ngtool` 非 XDG 标准细分目录

风险：

- 配置、备份、日志都放在同一根目录，需要清晰组织。

建议：

- 使用 `config.toml`、`backups/`、`logs/`、`cache/`、`tmp/` 子目录。
- 安装和卸载脚本明确说明该目录用途。

状态：用户已确认该路径。

### R18：MVP 暂不完整替代 Bash 脚本

风险：

- 早期 TUI 与现有 CLI 功能不完全一致。

建议：

- README 标注 TUI 开发状态。
- 保留 Bash 脚本作为稳定入口。
- 执行文档用 P0-P10 量化进度。

状态：用户已确认按 MVP 顺序推进。

### R20：aarch64 / 跨架构发布

风险：

- 目标部署环境为 OrangePi（aarch64），开发环境若为 x86_64 需要交叉编译。
- 部分 crate 可能依赖 C 库（如 `flate2` 的 zlib），交叉编译时缺工具链会失败。

建议：

- 在 P0 阶段先执行一次 `rustup target add aarch64-unknown-linux-gnu` 并空编译验证 target 可用。
- 优先选择纯 Rust 实现的 crate（`flate2` 默认 backend 已是纯 Rust）。
- 若需 C 库依赖，在 README 列出对应 sysroot 安装命令。
- 在 P10 阶段同时产出 `aarch64` 和 `x86_64` 两份 release。

状态：实现期由 P10-7、P10-8 闭环。

## 六、实施期间风险处理规则

开发过程中遇到风险时按以下规则处理：

1. 高风险项不得用"先这样"绕过，必须实现保护或降级。
2. 中风险项可以先用 MVP 方案，但必须在代码或文档中明确边界。
3. 低风险项不阻塞 MVP。
4. 如果风险会修改 `design.md` 的交互原则，先记录到本文档，不直接改设计文档。
5. 如果风险会影响数据安全或服务可用性，优先调整实现文档和架构文档。
