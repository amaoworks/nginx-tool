# Nginx-Tools TUI 实现执行文档

## 一、文档定位

本文档用于指导 `tui-next` 后续开发执行。执行计划必须贴合 `architecture.md`，并能量化项目进度、验收结果和剩余风险。

本文档不修改 `design.md`。交互设计以 `design.md` 为准，工程实现以 `architecture.md` 为准。

## 二、开发原则

1. 先打通架构闭环，再扩展页面。
2. 每个阶段必须有可运行产物。
3. 每个写操作必须有权限检查、错误展示和回滚策略。
4. 每个视图必须符合顶部标题栏、左侧边栏、右侧主视图、底部状态栏的布局约束。
5. 长任务必须异步执行，不能卡住 TUI。
6. 首版只支持中文、systemd、Debian/Ubuntu Nginx 目录结构。
7. 进度按验收项计算，不按代码行数计算。

## 三、总体里程碑

|------|------|------|------|------------------|----------|

| P4 | 站点列表与启停 | 列表、解析、启用、停用、回滚 | 12% | 4-5 | R1、R3、R15 |
| P5 | 新建站点 | 表单、模板、写入、测试、reload、回滚、简易证书 | 14% | 5-7 | R1、R3、R14 |

总进度 = 各阶段完成权重之和。阶段内部按验收项均分，除非该阶段另有说明。

> 权重总和 = 7+11+12+9+12+14+9+7+8+6+5 = **100%**
>
> 时间基线为单人开发的乐观估计，含调试和文档维护，不含外部联调。基线总计约 35-49 人天，约 7-10 周完成 MVP（P0-P7）+ 后续阶段。

## 四、P0 工程骨架

### 4.1 目标

创建可编译、可运行、可测试的 Rust 工程骨架。

### 4.2 入口条件

无前置依赖。文档（design / architecture / risks）已确认即可开工。

### 4.3 任务

| 编号 | 任务 | 验收标准 |
|------|------|----------|
| P0-1 | 创建 `tui-next/Cargo.toml` | `cargo check` 可运行 |
| P0-2 | 创建 `src/main.rs` | 启动后能打印版本或进入空 TUI |
| P0-3 | 添加核心依赖 | `ratatui`、`crossterm`、`tokio`、`clap`、`serde` 等可编译 |
| P0-4 | 建立模块目录 | `app`、`ui`、`domain`、`infra`、`template`、`config` 存在 |
| P0-5 | 添加基础 CI 命令约定 | 本地能运行 `cargo fmt`、`cargo clippy`、`cargo test` |
| P0-6 | 创建 `templates/` 占位 | `proxy.conf.j2`、`emby.conf.j2`、`static.conf.j2` 空骨架可被 `include_str!` 引用 |

### 4.4 完成定义

- `cargo check` 通过。
- `cargo test` 至少能运行空测试。
- 程序能启动并正常退出。
- 编译后的二进制启动 `--version` 显示版本号。

### 4.5 出口条件 / 下阶段入口

P1 可以开始：模块目录可用、Cargo workspace 编译通过。

### 4.6 风险闭环

无对应高风险项。

### 5.1 目标

### 5.2 入口条件

### 5.3 任务

| 编号 | 任务 | 验收标准 |
|------|------|----------|
| P1-1 | 初始化终端 | 进入 alternate screen，退出后恢复终端；`panic_hook` 能恢复终端 |
| P1-2 | 实现 root layout | 顶部 1 行、底部 1 行、左侧不小于 20 列 |
| P1-3 | 实现左侧菜单 | 仪表盘、站点管理、证书管理、日志查看、服务控制、备份还原 |
| P1-4 | 实现路由 | 上下键切换一级菜单，右侧视图变化 |
| P1-5 | 实现底部状态栏 | 根据当前路由显示不同快捷键 |
| P1-6 | 实现通知系统 | 成功 2 秒、失败 3 秒自动消失 |
| P1-7 | 实现确认弹窗 | 默认聚焦取消，Enter 执行，Esc 取消 |
| P1-8 | 实现覆盖模式框架 | 新建、编辑等只替换右侧主视图 |
| P1-9 | 信号处理 | `SIGWINCH` 触发重绘；`SIGINT`/`SIGTERM` 优雅退出并恢复终端（参见 architecture.md §17.1） |

### 5.4 完成定义

- `q` 可退出（软退出）。
- `Ctrl+C` 可硬退出且不留下乱码终端。
- `Esc` 在覆盖模式中返回上级。
- 弹窗不会用于普通编辑。
- 任意页面都保留顶部、左侧、底部结构。
- 终端 resize 后整屏重绘正常。

### 5.5 出口条件 / 下阶段入口

P2 可以开始：UI 框架可承载视图，事件循环稳定。

### 5.6 风险闭环

- R10 终端尺寸过小：通过 P1-10 验证。
- R11 快捷键冲突：底部状态栏按 Route 上下文显示，验收时检查每个 Route 的快捷键集合。
- R12 `Ctrl+Enter` 终端兼容性：在 P1-11 测试矩阵中加入 `Ctrl+Enter` 探测；不可用时记录并保留普通 `Enter` 提交路径。

## 六、P2 系统适配层

### 6.1 目标

建立所有系统交互的统一入口，避免 UI 直接操作系统。

### 6.2 入口条件

P1 出口条件满足。

### 6.3 任务

| 编号 | 任务 | 验收标准 |
|------|------|----------|
| P2-1 | 实现 `CommandExecutor` | 支持 program + args、stdout、stderr、exit code、超时、取消 |
| P2-2 | 实现权限检测 | root、只读参数、目录可写检测、目录可读检测 |
| P2-3 | 实现配置目录 | 自动创建 `~/.local/ngtool/{backups,logs,cache,tmp}` |
| P2-4 | 实现单实例软锁 | `~/.local/ngtool/tui.lock` 上 flock；冲突降级为只读（参见 architecture.md §15.0） |
| P2-5 | 实现 Nginx 目录扫描 | 可扫描 `sites-available` 与 `sites-enabled`；扫描失败降级显示环境异常 |
| P2-6 | 实现 systemd 适配 | `is-active`、`status --no-pager`、`reload`、`restart` |
| P2-7 | 实现 nginx 适配 | `nginx -v`、`nginx -t` |
| P2-8 | 实现 certbot 检测 | 缺失时返回可展示的依赖状态，不影响其他功能 |
| P2-9 | 实现操作审计日志 | `~/.local/ngtool/logs/audit.log` 写入 + 滚动（参见 architecture.md §18） |
| P2-10 | 实现临时目录回收 | 启动时清理 `tmp/` 中超过 7 天的残留 |

### 6.4 完成定义

- 非 root 启动进入只读模式。
- 缺少 certbot 不影响仪表盘和站点列表。
- 命令失败时 UI 能得到 stderr 和 exit code。
- 命令超时由执行器统一处理，不阻塞调用方。
- 写操作能追加到审计日志。

### 6.5 出口条件 / 下阶段入口

P3 可以开始：所有系统命令通过 executor 调用，权限模式和审计可用。

### 6.6 风险闭环

- R4 长任务阻塞：所有 executor 调用都是 async，验收时模拟 5 秒延时确认 UI 不冻结。
- R5 root/只读边界：通过 P2-2、P2-4 验证，UI 在切换模式时无残留状态。
- R8 systemd 兼容：缺 systemctl 时降级提示。

## 七、P3 仪表盘

### 7.1 目标

实现默认视图，替代 `nginx-monitor.sh` 的主要状态输出。

### 7.2 入口条件

P2 出口条件满足。

### 7.3 任务

| 编号 | 任务 | 验收标准 |
|------|------|----------|
| P3-1 | Nginx 运行状态 | 显示运行中/已停止/未知 |
| P3-2 | Nginx 版本 | 显示 `nginx -v` 解析结果 |
| P3-3 | 已启用站点数 | 从 `sites-enabled` 统计 |
| P3-4 | SSL 摘要 | 显示证书数量和临期状态；解析失败不影响其他项 |
| P3-5 | 系统资源 | 显示磁盘、内存 |
| P3-6 | 最近错误 | 显示最近 3 条 error.log |
| P3-7 | 手动刷新 | `r` 可刷新 |
| P3-8 | 自动刷新 | 默认 30 秒刷新（按 config.toml 可配置） |
| P3-9 | 并发采集与超时 | 七项并发执行，单命令 3s 超时，整次刷新 5s 上限（参见 architecture.md §11.2 性能与超时预算） |

### 7.4 完成定义

- 数据采集失败不会导致 TUI 崩溃。
- 刷新时保留旧数据直到新数据返回。
- 单项失败时该项显示 ⚠ 标记，其他项正常。
- 状态栏显示 `仪表盘 | [r] 刷新 | [q] 退出`。

### 7.5 出口条件 / 下阶段入口

P4 可以开始：仪表盘能验证数据采集链路稳定。

### 7.6 风险闭环

- R2 certbot 解析不稳定：通过 P3-4 单独失败不影响其他项验证。
- R4 长任务阻塞：通过 P3-9 并发+超时验证。

## 八、P4 站点列表与启停

### 8.1 目标

实现站点管理的列表视图和启用/停用操作。

### 8.2 入口条件

P3 出口条件满足。建议同时在 P4 之前完成 P7 的 `nginx -t` 和 `systemctl reload` 子能力（参见第二十节推进顺序），否则启停验收依赖会缺失。

### 8.3 任务

| 编号 | 任务 | 验收标准 |
|------|------|----------|
| P4-1 | 扫描站点配置 | 读取 `/etc/nginx/sites-available/*.conf` |
| P4-2 | 判断启用状态 | 根据 `sites-enabled` 符号链接判断 |
| P4-3 | 解析基础字段 | 解析 `server_name`、`proxy_pass`、`root` |
| P4-4 | 推断站点类型 | 代理、Emby、静态；Emby 识别基于工具标记或启发式 |
| P4-5 | 匹配 SSL 状态 | 根据域名匹配 certbot 输出；解析失败不影响列表 |
| P4-6 | 表格绘制 | 状态、名称、域名/目标、类型、SSL |
| P4-7 | 启用站点 | 创建符号链接、`nginx -t`、reload、失败回滚 |
| P4-8 | 停用站点 | 删除符号链接、`nginx -t`、reload、失败恢复 |
| P4-9 | 只读裁剪 | 非 root 时禁用启停 |
| P4-10 | 审计落盘 | 启用/停用成功或失败均写入 audit.log |

### 8.4 完成定义

- `↑↓` 可选择站点。
- `Enter` 可切换启用/停用。
- 启停失败时站点状态不被错误更新。
- 状态栏符合设计文档。

### 8.5 出口条件 / 下阶段入口

P5 可以开始：新建站点可以复用启停的 nginx -t/reload 路径。

### 8.6 风险闭环

- R1 写配置损坏：P4-7、P4-8 包含失败回滚验证。
- R3 注入槽误判：P4-4 类型推断对未知配置降级为"代理"，不影响原始配置编辑。
- R15 Emby 类型识别：见 P4-4 验收。

## 九、P5 新建站点

### 9.1 目标

实现新建站点覆盖模式，支持代理、Emby、静态站点，含简易证书申请。

### 9.2 入口条件

P4 出口条件满足。`certbot` 适配层（P2-8）已能检测可用性，但完整证书管理 UI 不在本阶段。

### 9.3 任务

| 编号 | 任务 | 验收标准 |
|------|------|----------|
| P5-1 | 表单状态模型 | 站点名、域名、类型、目标、启用、申请证书 |
| P5-2 | 焦点切换 | Tab、Shift+Tab、上下键、Space、Enter |
| P5-3 | 字段校验 | 必填、格式、重复站点、目标格式；"申请证书"必须依赖"立即启用"，否则禁用 |
| P5-4 | 快速提交 | `Ctrl+Enter` 提交但仍执行完整校验（终端不支持时降级为普通 `Enter`） |
| P5-5 | 模板渲染 | `proxy`、`emby`、`static` 模板通过 `include_str!` 编译期内嵌，输出包含规范注入槽标记 |
| P5-6 | 写入配置 | 写入 `/etc/nginx/sites-available/<name>.conf`，`O_CREAT \| O_EXCL` 防 race |
| P5-7 | 可选启用 | 创建符号链接、测试、reload |
| P5-8 | 可选证书（简易模式） | 仅当"立即启用"成功后调用 `certbot --nginx -d <server_name 全部域名>`；失败不回滚已写入的站点 |
| P5-9 | 失败回滚 | 按反向顺序删除链接、删除配置（参见 design.md "失败回滚机制"） |
| P5-10 | 放弃确认 | 表单有内容时 Esc 弹出确认 |
| P5-11 | 审计落盘 | site.create + 可能的 cert.request 各写一条 |

### 9.4 完成定义

- 新建成功后站点列表能看到新站点。
- 测试失败会回滚新建文件。
- 证书申请失败不会让站点回滚（已存在 server 配置可继续使用）。
- 只读模式不能进入可编辑表单。

### 9.5 出口条件 / 下阶段入口

P6 / P7 可以开始（彼此独立）。

### 9.6 风险闭环

- R1 写配置损坏：P5-9 反向回滚。
- R3 注入槽：P5-5 模板必须输出标准标记。
- R14 证书申请边界：P5-8 强制依赖立即启用，匹配 design.md 第 II 节。

## 十五、进度量化规则

### 15.1 阶段状态

每个任务状态只能是：

| 状态 | 含义 | 计分 |
|------|------|------|
| 未开始 | 未实现 | 0 |
| 开发中 | 有代码但不可验收 | 0 |
| 可验收 | 功能可运行但测试不足 | 0.7 |
| 已完成 | 验收标准和测试通过 | 1.0 |
| 暂缓 | 有明确原因延后 | 不计入当前阶段分母 |

### 15.2 计算方式

阶段完成度：

```text
阶段完成度 = 已完成任务分数之和 / 当前阶段计分任务数
```

总进度：

```text
总进度 = Σ(阶段完成度 × 阶段权重)
```

示例：

```text
P1 权重 12%
P1 共 8 个任务
其中 4 个完成，2 个可验收，2 个未开始
P1 完成度 = (4 * 1.0 + 2 * 0.7) / 8 = 67.5%
P1 对总进度贡献 = 12% * 67.5% = 8.1%
```

## 十九、MVP 验收标准

MVP 对应 P0-P7。MVP 范围内的"证书申请"专指 **P5-8 简易模式**：在新建站点流程内调用 `certbot --nginx`，不包括 P8 的完整证书管理页面。

MVP 完成必须满足：

1. 能启动 TUI 并正常退出（含 `q` 软退出和 `Ctrl+C` 硬退出）。
2. 中文界面完整可读，CJK 输入和长中文文本不破坏布局。
3. 仪表盘能展示 Nginx 状态，单项采集失败不影响整体。
4. 站点列表能展示现有配置。
5. root 模式能新建站点（含可选启用、可选简易证书申请）。
6. root 模式能启用和停用站点，含失败回滚。
7. 非 root 模式能浏览但不能写，UI 上写操作快捷键被禁用并有提示。
8. 服务控制能测试和重载配置，**不通过 shell `&&` 串联**。
9. 日志查看能实时跟踪。
10. 所有长任务期间 UI 不冻结，可被 `Esc` 取消或返回。
11. 所有写操作记录在审计日志中。
12. `SIGWINCH` 触发重绘不破坏当前路由和焦点。

## 二十、后续实现顺序建议

实际开发顺序与权重表的差异主要由依赖关系决定：

1. P0 工程骨架。
2. P1 全局布局、路由、信号处理。
3. P2 executor、配置目录、权限模式、单实例锁、审计日志。
4. P3 仪表盘，验证数据采集链路（并发 + 超时）。
5. **P7 部分**：先打通 P7-2 测试配置、P7-3 重载配置（不含 UI，仅 use case），作为后续阶段的共用依赖。
6. P4 站点列表与启停，复用 P7 的测试/重载能力。
7. P7 剩余：把 P7-1/4/5/6/7 在 UI 上集成。
8. P5 新建站点，依赖 P4/P7 的全部能力。
9. P6 日志查看，独立补齐实时任务模型，可与 P5 并行。
10. **此处达成 MVP 验收，可以发布 v0.1。**
11. P8 证书管理（在 P5-8 简易模式基础上扩展）。
12. P9 备份还原（含 manifest 和范围限定）。
13. P10 打磨发布（aarch64 交叉编译、release 物料）。

该顺序与权重表的主要差异是把 P7 的"测试/重载"子能力提前至 P4 之前，因为这是新建、编辑、启停、还原的共同依赖，必须尽早稳定。
